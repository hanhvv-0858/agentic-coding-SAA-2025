## Potential Challenges

### Technical Challenges

| Challenge | Impact | Proposed Solution |
|-----------|--------|-------------------|
| **Cloudflare Workers runtime incompatibilities** — some Node-only packages silently break | High | Use Workers-aware libs: `@supabase/ssr` (not `@supabase/auth-helpers-nextjs`), Web `crypto` (not `node:crypto`), Vitest (not Jest with jsdom) |
| **Workers cookie size limits** (Cloudflare Workers imposes a ~8 KB limit on a single `Cookie` header + `Set-Cookie`) | Medium | Supabase splits session cookies by default (`sb-<ref>-auth-token.0`, `.1`). Verify with a real deploy |
| **CSP for Google OAuth redirect** | Medium | CSP must allow `connect-src`, `form-action`, and `frame-src` for `accounts.google.com`. Decide Next.js middleware vs Cloudflare response headers (spec Q8) |
| **Open-redirect via `?next=`** | Medium (security) | Validate `next` against same-origin + no `//` authority + path-only; reject otherwise |
| **Font weight budget** — Montserrat 400 + 700 + Montserrat Alternates 700 ≈ 100–140 KB (woff2, subset latin+vietnamese) | Low | Load via `next/font/google` with `display: "swap"`, subset `["latin","vietnamese"]`, weights 400+700 only |
| **Hero background image weight** (key visual is a ~1440 × 1024 artwork) | Medium — affects LCP budget (SC-002 ≤ 2.5 s) | Ship as responsive `next/image` with AVIF + WebP fallback, priority-load, and an inline blur placeholder |

### Integration Challenges

| Challenge | Impact | Proposed Solution |
|-----------|--------|-------------------|
| **Domain allow-list must be enforced in two places** (Supabase config + callback re-check) | High (security) | Document exactly which domain(s) are allowed (spec Q1); add a unit test that exercises the callback with a disallowed domain |
| **Local dev against a real Supabase project vs. local `supabase start`** | Low | Use `supabase start` locally (seeded from `supabase/config.toml`); wire through `make up` |
| **`middleware.ts` for session refresh** — if dropped, the cookie TTL will silently expire | Medium | Add a session-refresh `middleware.ts` matching `/:path*` per the Supabase SSR guide; exclude `/_next/static`, `/favicon.ico`, `/images/*` |

## Recommendations

### Architecture Recommendations

1. **Vertical slice, not horizontal layers.** Build "a Sun* user can click Google and
   land on `/`" end-to-end first (US1 happy path), then layer on US2 (errors) and US3
   (language toggle). Avoids the trap of scaffolding a beautiful UI that doesn't
   authenticate.
2. **Split the page into Server + two Client islands.** `<LoginPage />` is a server
   component that owns the session check and the i18n copy. `<GoogleLoginButton />` and
   `<LanguageToggle />` are client components because each needs state + event handlers.
   Nothing else ships `"use client"`.
3. **Own the OAuth callback route.** Don't rely on the default Supabase URL; create
   `src/app/auth/callback/route.ts`. Running under our domain lets us enforce the
   allow-list, validate the `next` param, and set cookies in the Workers runtime.
4. **Static-first imagery.** Both the full-page background and the "ROOT FURTHER" hero
   ship as `public/images/*` served with `Cache-Control: public, max-age=31536000,
   immutable` — not fetched at runtime.
5. **Establish `<Icon />` early.** Every feature after Login needs it; introducing it
   in the same PR prevents drift.

### Implementation Recommendations

1. **Start with**: Phase 0 (constitution alignment) and Phase 1 (foundation). These are
   prerequisites for *every* spec, not just Login — so the investment pays back across
   the whole project.
2. **Leverage**: nothing yet exists, so the Login PR becomes the architectural template
   for subsequent PRs. Review it with that lens.
3. **Avoid**: `useEffect`-based session reads, raw `<svg>` tags, hard-coded Vietnamese
   copy inside JSX, any Node-only import inside the middleware or the callback handler.

### Testing Recommendations

1. **Focus on**: the `/auth/callback` route — it's the one place where auth logic
   crosses a trust boundary. It must have unit tests for (a) valid Sun\* domain, (b)
   disallowed domain, (c) `exchangeCodeForSession` failure, (d) invalid `next` param,
   and (e) cookie-blocked browser.
2. **Mock**: Supabase client methods (`signInWithOAuth`, `exchangeCodeForSession`,
   `getUser`). Do NOT mock `fetch` globally — Vitest's `vi.mock` on the Supabase module
   is cleaner.
3. **E2E scenarios**: one Playwright happy-path (`/login` → click CTA → mocked OAuth
   success → `/`), one denied-consent path, one already-signed-in path (session cookie
   present → 302). Gated behind a mock Supabase project so the suite can run offline in
   CI.
